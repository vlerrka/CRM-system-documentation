Функциональные требования

UC-1​

IDUC - 01
НазваниеЗапись на курс
АкторыУченик (потенциальный), менеджер
ПредусловияМенеджер авторизован в системе. Контакт уже создан в системе (или создается в процессе) с пометкой "потенциальный клиент". Курс активен и доступен для записи
Основной поток1. Менеджер получает обращение (заявка с сайта, звонок, сообщение в мессенджере)
  1. Менеджер проверяет, есть ли контакт в системе 2а. Контакта нет в системе, менеджер заносит данные, контакты для связи, историю коммуникаций
  2. Менеджер связывается с потенциальным учеником (звонок, Telegram), консультирует по курсу, отвечает на вопросы
  3. После согласия на запись менеджер нажимает кнопку "Записать на курс" в карточке контакта
  4. Система открывает окно выбора курса, группы (если есть несколько потоков), тарифа (с куратором / без, с проверкой ДЗ / без)
  5. Менеджер выбирает нужные параметры и нажимает "Продолжить"
  6. Система проверяет наличие мест в группе (если есть ограничения)
  7. Если мест нет, система предлагает альтернативы (другая группа, лист ожидания, следующий поток)
  8. Менеджер согласовывает с потенциальным учеником способ оплаты 9а. Если оплата сейчас: менеджер отправляет ссылку на оплату (система генерирует платежную ссылку) или принимает оплату по терминалу (в офисе) 9б. Если оплата позже или рассрочка: менеджер выставляет счет с отложенным платежом или активирует курс в кредит/рассрочку
  9. После подтверждения оплаты (или активации рассрочки) система: меняет статус контакта с "потенциальный" на "ученик", добавляет контакт в выбранную группу, открывает доступ к материалам курса, отправляет приветственное письмо с доступом, сохраняет историю оплаты в карточке контакта
  10. Менеджер видит уведомление "Контакт успешно записан на курс" и может добавить комментарий в карточку| |Постусловия|Контакт получает статус "ученик", зачислен на курс, получил доступ к материалам, информация об оплате зафиксирована| |Альтернативные потоки|4а. Если потенциальный ученик отказался, менеджер выбирает причину отказа в выпадающем списке, контакт остается в статусе "потенциальный", но помечается как "отказ" для будущего анализа 8а. Если в группе нет мест, система автоматически предлагает ближайший доступный поток с похожими параметрами 10а1. При отправке платежной ссылки система фиксирует факт отправки и переводит контакт в статус "ожидание оплаты" (промежуточный статус| |Исключительные потоки|Технический сбой при генерации платежной ссылки - менеджер получает уведомление и может отправить ссылку вручную из шаблона или выставить счет через email Курс был закрыт или удален во время обработки - Система проверяет актуальность при нажатии "Записать" и выводит сообщение об ошибке с предложением альтернатив Оплата не подтверждается более 24 часов - Система напоминает менеджеру связаться с контактом для уточнения статуса| |Бизнес правила|BR 01.1 - Система должна обеспечивать единую карточку контакта на всем жизненном цикле BR 01.2 - Система должна фиксировать все коммуникации с контактом в хронологическом порядке|

UC-2​

IDUC - 02
НазваниеВедение электронного журнала успеваемости
АкторыКуратор (редактирование), Система, Руководитель и преподаватель(просмотр)
ПредусловияКуратор авторизован в системе. У него есть доступ к одной или нескольким группам с соответствующими правами. В группе есть ученики, назначены занятия и домашние задания
Основной поток1. Куратор заходит в раздел "Журнал успеваемости"
  1. Система отображает список групп, закрепленных за куратором, с краткой статистикой (количество учеников, дата последнего обновления журнала, количество непроверенных ДЗ)
  2. Система открывает журнал группы в табличном виде:
  • строки: ученики группы (отсортированы по алфавиту или по желанию куратора)
  • столбцы сгруппированы по датам занятий, внутри каждой даты: "Посещаемость", "ДЗ"
  • дополнительные колонки: средний балл ученика, общее количество пропусков
  1. Куратор видит, какие занятия уже прошли (даты выделены), какие ДЗ ожидают проверки (помечены значком)
  2. Куратор начинает заполнение журнала по прошедшему занятию: был/не был, оценка за домашнее
  3. При необходимости куратор добавляет комментарий к любой оценке (например, "домашнее задание сдано после дедлайна", "уважительная причина пропуска")
  4. Куратор может заполнять ячейки последовательно, перемещаясь по таблице с помощью клавиш Enter, Tab или стрелок
  5. Система автоматически сохраняет каждую введенную отметку (или сохраняет при переходе к следующей ячейке)
  6. По мере заполнения система пересчитывает: средний балл каждого ученика (с учетом весов оценок, если настроены), количество пропусков, процент выполненных ДЗ
  7. Система позволяет просмотр журнала преподавателем, руководителем и другими пользователями. Ученикам доступна информация только по их успеваемости| |Постусловия|Все отметки по группе за определенную дату/задание сохранены, данные доступны для аналитики, ученики видят свои результаты в личном кабинете| |Альтернативные потоки|4а. Куратор может переключить отображение журнала в другой вид: "по ученикам" (вся история одного ученика) или "по ДЗ" (сводка по домашним заданиям), либо за определенный период 5а. Куратор может использовать шаблоны заполнения: например, отметить всех учеников как присутствующих одной кнопкой, а затем исправить отсутствующих| |Исключительные потоки|Куратор пытается выставить оценку за занятие, которое еще не прошло - Система блокирует ввод и выводит сообщение "Нельзя выставлять оценки за будущие даты" Ученик отчислен из группы - Система отображает его в журнале серым цветом с пометкой "отчислен", новые оценки ставить нельзя, старые сохраняются| |Бизнес правила|BR 02.1 - Система должна предоставлять руководителю возможность просматривать любой журнал без редактирования BR 02.2 - Система должна предоставлять единый интерфейс для учета посещаемости и успеваемости в одном журнале BR 02.3 - Система должна предотвращать выставление оценок за будущие даты|

UC-3​

IDUC - 03
НазваниеСоздание и наполнение курса учебными материалами
АкторыПреподаватель (автор курса), Система
ПредусловияПреподаватель авторизован в системе. У него есть права на создание и редактирование курсов
Основной поток1. Преподаватель заходит в раздел "Мои курсы" и нажимает "Создать новый курс"
  1. Система открывает форму создания курса с полями: название, описание, обложка, категория, предмет
  2. Преподаватель заполняет информацию и нажимает "Сохранить и перейти к наполнению"
  3. Система создает курс и открывает интерфейс конструктора
  4. В конструкторе преподаватель нажимает "Добавить модуль", вводит название
  5. Внутри модуля нажимает "Добавить урок", заполняет: название, тип урока (видео/текст/практика), продолжительность, описание
  6. В блоке "Материалы урока" преподаватель нажимает "Добавить материал", выбирает тип (видео, презентация, документ, ссылка), загружает файл или вставляет ссылку, добавляет описание и сохраняет
  7. В блоке "Домашнее задание" преподаватель нажимает "Добавить", заполняет: название, описание, тип выполнения, дедлайн, максимальный балл, и сохраняет задание
  8. Преподаватель повторяет шаги 6-8 для создания всех уроков модуля, затем шаги 5-8 для создания следующих модулей
  9. Преподаватель при необходимости перетаскивает уроки и модули мышкой, меняя их порядок; система автоматически сохраняет изменения
  10. Преподаватель переходит в раздел "Настройки доступа", выбирает вариант открытия уроков (все сразу, по расписанию, по выполнению ДЗ) и настраивает параметры
  11. Преподаватель нажимает "Предпросмотр", проверяет, как курс будет выглядеть для учеников
  12. Преподаватель нажимает "Опубликовать курс"
  13. Система проверяет заполнение всех обязательных полей и наличие материалов
  14. Если проверка пройдена, система меняет статус курса на "Опубликован" и уведомляет об успешной публикации| |Постусловия|Курс создан, наполнен материалами, структурирован по модулям и урокам, опубликован и доступен для назначения ученикам| |Альтернативные потоки|1а. Если курс уже существует, Преподаватель на шаге 1 выбирает курс из списка "Мои курсы", нажимает "Редактировать" и переходит к шагу 4 (конструктор), где может изменять любые элементы: добавлять/удалять модули и уроки, менять материалы, обновлять настройки 13а. Если преподаватель не завершил наполнение и вышел, система сохраняет черновик; при следующем входе предлагает продолжить с того же места 13б. Преподаватель может отменить все изменения, сделанные с момента последнего сохранения, через кнопку "Отменить"| |Исключительные потоки|Преподаватель пытается опубликовать курс без материалов в уроках - Система предупреждает и показывает список пустых уроков Загружается файл неподдерживаемого формата - Система отклоняет загрузку и показывает список допустимых форматов Технический сбой при загрузке - Система сохраняет черновик и предлагает повторить позже| |Бизнес правила|BR 03.1 - Интуитивно понятный конструктор курса без программирования BR 03.2 - Хранение материалов в облаке с быстрым доступом|

UC-5​

IDUC - 05
НазваниеАналитика и выявление «спящих» учеников
АкторыСистема, Куратор, Менеджер (руководитель отдела продаж или заботы)
ПредусловияВ системе настроены правила отнесения ученика к категории «спящий» (пороги неактивности: количество дней без входа, пропущенных ДЗ, отсутствия на занятиях). Ученики имеют статус «активный» или «в обучении»
Основной поток1. В заданное время (например, 8:00 ежедневно) система запускает фоновый процесс проверки активности всех учеников, имеющих статус «активный» или «в обучении»
  1. Для каждого ученика система анализирует следующие параметры (согласно настройкам школы): количество дней с последнего входа в личный кабинет, количество пропущенных занятий подряд, количество несданных домашних заданий (просроченных)
  2. Система сравнивает фактические показатели с пороговыми значениями, заданными в настройках
  3. Если ученик попадает под критерии «спящего», система изменяет его статус на «спящий» (или добавляет соответствующую метку в карточку)
  4. Система фиксирует дату перехода в статус «спящий» и причину (какой критерий сработал)
  5. Система добавляет ученика в динамический сегмент «Спящие» для дальнейшей работы
  6. Система определяет ответственного куратора (из настроек группы или курса)
  7. Система отправляет куратору уведомление (в CRM, Telegram, email) со списком новых «спящих» учеников, требующих внимания, и краткой информацией по каждому (когда последний раз заходил, какие ДЗ пропустил)
  8. В карточке каждого такого ученика система автоматически создает задачу для куратора: «Связаться с учеником, выяснить причины отсутствия активности» с указанием приоритета и дедлайна (например, 24 часа)
  9. Менеджер заходит в раздел «Аналитика» и выбирает блок «Спящие ученики»
  10. Система отображает дашборд с ключевыми метриками: общее количество «спящих» на текущий момент, динамика появления «спящих» по неделям/месяцам, основные причины «засыпания» (на основе данных: пропуски, отсутствие входов и т.д.)| |Постусловия|Сформирован список «спящих» учеников, ответственные кураторы получили уведомления, запущены сценарии возврата (напоминания, персональные сообщения). Менеджер видит общую статистику по процессу| |Альтернативные потоки|4а. Некоторые ученики (например, в академическом отпуске) могут быть исключены из автоматической проверки 4б. Куратор или менеджер может снять статус «спящий» вручную| |Исключительные потоки|Сбой при отправке уведомлений - Система сохраняет список в журнале и повторяет отправку через час Конфликт критериев (ученик попадает под несколько)- Система фиксирует первое сработавшее условие как причину| |Бизнес правила|BR 05.1 - Система должна предоставлять возможность менеджерам изменять критерии "спящих" BR 05.2 - Система должна предоставлять менеджеру аналитику по «спящим» и эффективности их возврата|
